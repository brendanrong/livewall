# LiveWall — Project Handoff

> **For an AI assistant (or human) picking this up:** read this file first, then `audit-report.md`, then skim `Sources/`. Everything you need to be productive is captured here. The project is small (~3k LOC of Swift across 15 files) and self-contained.

---

## What this is

LiveWall is a **native macOS menu-bar app that turns the desktop into a video wallpaper**. Single-file mode, folder-with-rotation mode, or a web URL (YouTube auto-converts to a muted/looping embed). Built primarily for OLED displays — the goal is keeping pixels moving so they don't burn in.

Brendan (the owner) uses it on a MacBook Pro with a built-in Retina display, and is the only user.

It's a hand-rolled Swift app — **no Xcode project**, just `.swift` files compiled directly with `swiftc` via `build.sh`. This is intentional: keeps the project small, easy to read, easy to ship.

## Build / run / package

All from inside `LiveWall/`:

```bash
./build.sh              # compiles Sources/*.swift → LiveWall.app
./make_dmg.sh           # builds (if needed) and packages → LiveWall.dmg
```

Requires Xcode Command Line Tools (`xcode-select --install`).

To launch after building:

```bash
open ./LiveWall.app
# Then click the layered-rectangles + play icon in the menu bar.
```

To install for real: drag `LiveWall.app` to `/Applications/` (Launch at Login expects it there).

The build is **ad-hoc codesigned** today (`build.sh:78`), so first launch shows the "cannot verify developer" Gatekeeper warning — right-click → Open the first time. See "Distribution work" below for the proper fix.

## Architecture, briefly

```
                 main.swift
                     │
                     ▼
              AppDelegate ───────► HotkeyManager ──► (⌘⌥P fires controller.toggleEnabled)
                  │  │ │ │
                  │  │ │ └──────► BatteryMonitor ──┐
                  │  │ └────────► FullscreenMonitor ┴──► refreshPowerPause() ─► controller.setPausedByPower()
                  │  └──────────► StatusMenu ◄─────── Preferences (singleton)
                  │                  ▲
                  ▼                  │
        WallpaperController ─────────┘
            │      │
            │      └─► PreferencesWindowController (lazy: shown on demand)
            ▼
        [WallpaperWindow] (one per target display)
            │
            ├─► VideoWallpaperView (AVQueuePlayer + AVPlayerLooper)
            └─► WKWebView (for URL sources)
```

**Source resolution per display:** the controller maintains a global source (file/folder/URL) and an optional per-display override map. `resolveSource(for: displayID)` returns the override if present, else the global. Folder mode is global-only (rotates one playlist across all folder-mode windows synchronously).

**Power-aware pausing:** if `pauseOnBattery && onBattery` OR `pauseOnFullscreen && anyFullscreenAppDetected`, all video views are paused. State changes flow through `AppDelegate.refreshPowerPause()`.

**Master enable/disable:** `Preferences.wallpaperEnabled` is the single source of truth. The toggle on Settings (General), the status menu item, and the global hotkey all flip it. Changes post `WallpaperController.enabledStateChangedNotification` so the other surfaces refresh.

## What Brendan cares about (preferences captured over the iteration)

These came up during development and aren't necessarily obvious from the code:

1. **The menu bar icon is final.** It's the SF Symbol `play.rectangle.on.rectangle.fill` (with a slash variant when disabled). Do not redesign it. The custom brick-wall PNGs in `Resources/MenuBarIcon*.png` are unused legacy — keep them around for reference but don't wire them back in.

2. **The app icon is `Resources/livewall app icon.png`** (orange/red gradient with stacked white video frames). Source of truth. The script `make_app_icon_from_source.py` (in the outputs folder when developing) just resizes it into the iconset. Do not regenerate from PIL — Brendan likes this exact icon.

3. **Per-display source intentionally only supports File and URL** — not folder. Folders stay global. This was a scope decision; rotating different folders on different displays in lockstep is fine, but each display rotating its OWN folder independently was deferred.

4. **The brick wall texture in earlier icon iterations was problematic.** Brendan went through several rounds — too prominent, too subtle, then "actually I want a different design entirely". Final answer: orange/red gradient icon, not the brick-wall design. Do not bring back bricks.

5. **Support / feedback links live at the top of `PreferencesWindow.swift`**:
   - `SUPPORT_URL = "https://www.buymeacoffee.com/brendanbong"`
   - `FEEDBACK_TO = "brendanbong22@gmail.com"`

6. **No external dependencies.** The project deliberately doesn't use Sparkle, MASShortcut, or any third-party framework. Adding one is a significant decision, not a casual change.

7. **No Xcode project file.** Don't create one. The shell-script build is the build system.

## Deferred / known issues

### Functional features deferred (Brendan asked for them but explicitly chose to defer)

- **Sparkle auto-update.** Needs hosted `appcast.xml` + ed25519 signing key. Code-side it's `import Sparkle`, `SPUUpdater.startUpdater()`, an `SUFeedURL` Info.plist key. Realistically can't ship until the project has a domain to host appcast on AND Brendan has a Developer ID cert (Sparkle refuses to update ad-hoc-signed apps).
- **Notarisation.** `xcrun notarytool submit LiveWall.zip --apple-id … --team-id … --password $APP_PWD --wait`. Needs Developer ID cert + app-specific password. Easiest as a `notarize.sh` script. Without this, every user has to right-click → Open on first launch.

### Audit findings (read `audit-report.md` for full detail)

The audit found 1 real bug worth fixing immediately:

- **B-1**: `Preferences.swift:137` — `hotkeyKeyCode` getter rejects `kVK_ANSI_A` (== 0). If a user records ⌘⌥A, it silently reverts to ⌘⌥P next launch.

Plus medium-severity items: a cross-fade race on rapid `play()` calls, an over-broad `NSAllowsArbitraryLoads` ATS exception, the WKWebView running with full JS for arbitrary URLs, and the ad-hoc signing situation. See the audit doc for file:line refs.

The biggest tech debt: `PreferencesWindow.swift` is 1264 lines. Splitting per-pane is the recommended next refactor before adding more sections.
